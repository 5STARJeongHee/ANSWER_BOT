## 2. 문제 정의

### 현재 상황 (As-Is)

- 사내 Slack 채널에서 직원들이 동일하거나 유사한 질문을 반복적으로 올리고 있음.
- 담당자가 매번 수동으로 답변해야 하여 업무 중단이 발생함.
- 무료 Slack 플랜으로 인해 90일 이후 메시지가 검색 불가능해지므로, 과거 맥락이 소실됨.
- 새 입사자가 온보딩 중 기본적인 업무 정보를 찾는 데 과도한 시간이 소요됨.

### 해결해야 할 문제

- 반복 질문에 대한 자동 응답으로 담당자 업무 부담을 줄여야 함.
- 사라지는 Slack 대화 이력을 별도 저장소에 누적하여 RAG(Retrieval-Augmented Generation) 기반 답변 품질을 높여야 함.
- LLM이 답변하기 어려운 케이스는 적절한 담당자에게 escalation해야 함.

### 왜 지금 해결해야 하는가

- 조직 성장으로 Slack 내 반복 질문 빈도가 증가하고 있으며, 무료 플랜의 90일 메시지 소실 문제로 인한 업무 컨텍스트 손실이 실질적인 비용으로 전환되고 있음.
- 무료 OpenRouter 모델(nex-agi/nex-n2-pro:free)의 등장으로 추가 LLM 비용 없이 구현 가능한 시점이 됨.

## 3. 목표 및 성공 지표

### 제품 목표

1. 반복 질문에 대한 자동 답변률 70% 이상 달성하여 담당자 수동 응답 부담을 감소시킴.
2. 과거 대화 이력을 DB에 누적하여 90일 이후에도 사내 지식 기반(Knowledge Base)을 유지함.
3. 챗봇이 답변 불가 판단 시 적절한 담당자에게 fallback하여 사용자 불만을 최소화함.

### 성공 지표 (KPI)

| 지표 | 목표값 | 측정 주기 |
|------|--------|---------|
| 자동 답변 처리율 | 전체 감지 질문의 70% 이상 | 주간 |
| 사용자 만족도 (thumbs up/down) | 긍정 응답 비율 75% 이상 | 월간 |
| 평균 응답 시간 | ack 3초 이내, 최종 답변 15초 이내 | 실시간 모니터링 |
| 챗봇 가용성 | 월간 99% 이상 (uptime) | 월간 |
| 대화 누적 보존율 | 수집 대상 메시지 대비 99% 이상 저장 | 일간 |
| Fallback 정확도 | 불확실 답변에서 fallback 발동률 90% 이상 | 주간 |

### 비즈니스 임팩트

- 담당자 1인당 주간 반복 질문 응답 시간 절감 (목표: 2시간/주 절감).
- 신규 입사자 온보딩 셀프서비스화로 HR 부담 감소.
- 사내 지식 자산의 체계적 누적 및 재활용.

## 4. 타겟 사용자

### 주요 페르소나

#### P1. 업무 질문자 (개발자 / 비개발자)
- **특성**: 특정 업무 절차, 시스템 사용법, 담당자 정보 등을 Slack에서 실시간으로 묻는 직원.
- **채널**: 전사 공용 채널, 팀 채널, DM.
- **Pain Point**: 답변을 기다리는 시간 낭비, 같은 질문을 반복하는 것에 대한 피로감.

#### P2. 답변 담당자 (시니어 직원 / 총무 / IT 관리자)
- **특성**: 반복 질문을 가장 많이 받는 조직 내 전문가.
- **Pain Point**: 동일 질문에 대한 반복 응답으로 집중 업무 시간 침해.

#### P3. 신규 입사자
- **특성**: 온보딩 중 기본 사내 정보를 모르는 상태.
- **Pain Point**: 누구에게 물어봐야 할지 모르고, 물어보기 민망한 질문도 많음.

### 사용자 여정 (User Journey) 요약

```
직원이 Slack 채널에 질문 작성
    → 챗봇이 메시지 감지 및 질문 여부 분류
    → 관련 과거 대화를 RAG 검색
    → LLM으로 답변 생성
    → Slack 스레드에 자동 회신 (3초 내 ack → 15초 내 최종 답변)
    → 불확실 케이스는 담당자 호출 메시지로 fallback
    → 답변 이력을 DB에 저장하여 차후 RAG 재활용
```

## 5. 기능 요구사항

### MVP - Phase 1 (필수 기능, 6~8주)

#### F-01. 메시지 이벤트 수신 및 질문 감지
- **설명**: Slack Socket Mode로 채널 메시지와 앱 멘션(`@챗봇`) 이벤트를 실시간 수신함. 질문/요청 여부를 규칙 기반 + LLM 보조 분류기로 판별함.
- **우선순위**: Must Have.
- **수락 기준 (Acceptance Criteria)**:
  - `app_mention` 이벤트와 지정 채널의 `message` 이벤트가 정상 수신됨.
  - 분류기가 질문(QUESTION), 요청(REQUEST), 해당 없음(NONE)을 JSON 형식으로 반환함.
  - 3초 이내에 Slack에 ack 응답이 전송됨.

#### F-02. RAG 기반 컨텍스트 검색 및 답변 생성
- **설명**: PostgreSQL + pgvector에 저장된 과거 대화 임베딩에서 현재 질문과 유사한 컨텍스트를 검색(코사인 유사도)하여 LLM 프롬프트에 주입하고 답변을 생성함.
- **우선순위**: Must Have.
- **수락 기준**:
  - 질문 임베딩 벡터로 pgvector에서 상위 5개 관련 청크를 검색함.
  - 검색된 컨텍스트 + 최근 5개 메시지를 프롬프트에 주입하여 LLM(nex-agi/nex-n2-pro:free via OpenRouter) 호출함.
  - 최종 답변이 Slack 스레드에 15초 이내에 게시됨.
  - 컨텍스트 없는 경우에도 LLM이 기본 답변을 생성함.

#### F-03. 실시간 대화 수집 및 저장
- **설명**: 수신되는 모든 메시지(봇 응답 포함)를 `conversation_message` 테이블에 실시간으로 적재하고, 임베딩을 생성하여 `context_embedding` 테이블에 저장함.
- **우선순위**: Must Have.
- **수락 기준**:
  - 이벤트 수신 시 DB에 메시지가 저장됨 (중복 방지: `event_id` unique 체크).
  - sentence-transformers 임베딩이 비동기로 생성 및 저장됨.
  - PII 필터링(이메일, 전화번호 마스킹)이 저장 전 적용됨.

#### F-04. 무료 Slack 90일 이력 백필(Backfill)
- **설명**: 챗봇 최초 배포 시 `conversations.history` API를 사용해 채널별 최근 90일 대화를 일괄 수집하여 DB에 적재함.
- **우선순위**: Must Have.
- **수락 기준**:
  - 지정 채널의 최근 90일 메시지가 배치 실행으로 수집됨.
  - 이미 수집된 메시지는 중복 삽입 없이 건너뜀.
  - 백필 완료 후 로그로 수집 건수가 기록됨.

#### F-05. 담당자 호출 Fallback
- **설명**: LLM 답변 신뢰도가 낮거나 사실 확인이 필요한 경우, 답변 생성 대신 담당자 멘션 또는 안내 메시지를 Slack에 전송함.
- **우선순위**: Must Have.
- **수락 기준**:
  - Fallback 판단 프롬프트 평가 결과 `can_answer_directly: false`이면 담당자 호출 메시지가 전송됨.
  - Fallback 메시지에는 "어떤 채널/담당자에게 문의하세요" 안내가 포함됨.
  - Fallback 발동 이력이 DB에 기록됨.

#### F-06. Docker 기반 배포
- **설명**: 전체 서비스(Python 앱 + PostgreSQL + pgvector)를 `docker-compose.yml`로 단일 명령으로 기동 가능하게 함.
- **우선순위**: Must Have.
- **수락 기준**:
  - `docker compose up -d`로 전체 스택이 정상 기동됨.
  - `.env` 파일로 시크릿을 주입하고, 이미지에 하드코딩된 시크릿이 없음.
  - 컨테이너 재시작 시 DB 데이터가 유지됨 (볼륨 마운트).

## 6. 비기능 요구사항

### 성능 목표

| 항목 | 목표 |
|------|------|
| Slack ack 응답 시간 | 3초 이내 (Slack 이벤트 타임아웃 방지) |
| 최종 답변 전송 시간 | 질문 수신 후 15초 이내 |
| RAG 검색 응답 시간 | 단일 쿼리 기준 1초 이내 |
| DB 쓰기 처리량 | 초당 50건 이상 (이벤트 동시성 대응) |

### 가용성 및 신뢰성

- 서비스 가용성(Uptime) 목표: 월간 99% 이상.
- Slack API 재시도 로직: 지수 백오프(Exponential Backoff) 적용, 최대 3회 재시도.
- LLM API 오류 시: fallback 메시지 자동 전송 (서비스 무중단 유지).
- DB 연결 실패 시: 메시지 수신은 계속하고 큐 또는 로컬 파일에 임시 저장 후 재적재.

### 보안 요구사항

- API Key(OPENROUTER_API_KEY, SLACK_BOT_TOKEN 등)는 `.env` 파일로 관리하며 코드/레포에 절대 포함하지 않음.
- Slack 이벤트 서명 검증(Signing Secret)을 Bolt 프레임워크 기본 기능으로 적용함.
- PII(개인정보) 필터링: 메시지 저장 전 이메일, 전화번호 등을 마스킹 처리함.
- DB는 Docker 내부 네트워크에서만 접근 가능하도록 포트 노출 최소화.

### 확장성

- 처음에는 단일 VM(또는 로컬 서버)에 배포하며, 향후 수평 확장 시 상태를 DB에만 의존하도록 설계.
- 채널 추가 시 설정 파일 수정만으로 운영 가능하도록 하드코딩 지양.

### 접근성 및 UX

- 봇 응답은 한국어로 작성하며, 2~5문장 내 핵심 전달.
- 답변 불가 상황에서도 빈 응답 없이 반드시 안내 메시지 전송.
- Slack 스레드 회신 방식으로 채널 채팅 흐름 방해 최소화.

## 8. 사용자 스토리

### US-01. 자동 답변 수신

> As a **사내 직원 (질문자)**, I want to **Slack 채널에 질문을 올리면 챗봇이 자동으로 관련 답변을 달아주기를**, so that **담당자 응답을 기다리지 않고 즉시 업무를 진행할 수 있다.**

**수락 기준.**
- 질문 감지 후 3초 이내에 "답변 중..." 메시지 또는 최종 답변이 스레드에 게시됨.
- 답변은 과거 관련 대화를 기반으로 생성됨.
- 출처 없는 추측 답변 시 "확인이 필요합니다"라고 명시됨.

### US-02. Fallback 안내 수신

> As a **사내 직원 (질문자)**, I want to **챗봇이 답변하기 어려운 질문에서 담당자를 안내해주기를**, so that **헛돌지 않고 정확한 담당자에게 빠르게 연락할 수 있다.**

**수락 기준.**
- Fallback 발동 시 담당자 멘션 또는 "XX 담당자에게 문의하세요" 메시지가 전송됨.
- Fallback 메시지에 빈 본문이 없음.

### US-03. 반복 질문 부담 감소

> As a **담당자 (시니어 직원)**, I want to **반복 질문을 챗봇이 처리해주기를**, so that **동일 질문 응답에 소비하는 시간을 줄이고 핵심 업무에 집중할 수 있다.**

**수락 기준.**
- 이전에 답변된 적 있는 질문은 RAG를 통해 자동 처리됨.
- 챗봇이 답변한 경우 담당자 노티피케이션이 발생하지 않음.

### US-04. 과거 대화 활용 답변

> As a **챗봇 시스템**, I want to **90일 이전 대화도 DB에서 검색하여 답변에 활용하기를**, so that **무료 Slack 플랜의 메시지 소실 제약을 극복할 수 있다.**

**수락 기준.**
- 챗봇 배포 후 90일이 지난 시점에도 과거 DB에서 관련 컨텍스트가 검색됨.
- DB에 저장된 메시지는 Slack 채널 삭제/소실 이후에도 유지됨.

### US-05. 신규 입사자 온보딩 자조(Self-Service)

> As a **신규 입사자**, I want to **기본 업무 정보(회의실 예약 방법, 인사 시스템 위치 등)를 챗봇에게 물어보기를**, so that **선배에게 부담을 주지 않고 스스로 온보딩을 진행할 수 있다.**

**수락 기준.**
- 일반적 온보딩 질문(예: "회의실 예약 어떻게 해요?")에 대해 챗봇이 관련 과거 대화 기반으로 답변함.
- 정보 없으면 담당 채널 또는 담당자로 안내함.

## 10. 개발 일정 (Roadmap)

| Phase | 기간 | 주요 산출물 | 담당 |
|-------|------|------------|------|
| Phase 0: 환경 설정 | 1주차 | Slack App 생성, OAuth Scope 설정, DB 스키마 확정, Docker 기본 환경 구성 | 개발팀 |
| Phase 1: Slack 연동 + 이벤트 수신 | 2주차 | event_handler.py, classifier.py, Slack echo 테스트 완료 | 개발팀 |
| Phase 2: DB 수집 파이프라인 | 3~4주차 | collector.py, models.py, 백필 배치, PII 필터 구현 | 개발팀 |
| Phase 3: RAG + 답변 생성 | 5~6주차 | context_retriever.py, llm_service.py, fallback 로직, 프롬프트 완성 | 개발팀 |
| Phase 4: 통합 테스트 + 배포 | 7~8주차 | docker-compose 완성, 파일럿 채널 배포, 모니터링 구성 | 개발팀 + 운영팀 |
| Phase 5: V2 - 요약 배치 | 9~10주차 | summarizer.py, APScheduler 등록, 요약 기반 RAG 전환 | 개발팀 |
| Phase 6: V2 - 피드백 수집 | 11~12주차 | 이모지 피드백 수집, 대시보드 기반 데이터 분석 | 개발팀 + PM |

> 버퍼: 각 Phase당 15% 버퍼(작업일 기준) 포함. 단, Phase 0~4는 MVP 기준 8주 이내 완료 목표.

### 각 Phase 완료 기준 (Definition of Done)

- **Phase 0**: `docker compose up -d`로 PostgreSQL + pgvector 정상 기동. Slack App이 Socket Mode로 연결됨.
- **Phase 1**: 지정 채널에서 멘션 메시지 수신 시 3초 이내 echo 응답이 스레드로 전송됨.
- **Phase 2**: 90일 백필 배치 실행 후 대화 메시지 및 임베딩이 DB에 저장됨. 중복 방지 확인됨.
- **Phase 3**: 질문 메시지에 대해 RAG 컨텍스트 기반 LLM 답변이 15초 이내에 스레드로 전송됨. Fallback 동작 확인됨.
- **Phase 4**: 파일럿 채널 2주 운영 후 자동 답변률 60% 이상, 오류율 1% 미만.
- **Phase 5**: 주간 요약 배치가 APScheduler로 자동 실행되고 `context_summary` 테이블에 저장됨.
- **Phase 6**: 답변 메시지에 thumbs up/down 반응이 기록되고 통계 조회 가능함.

## 11. 의존성 및 리스크

### 기술적 의존성

| 의존성 | 설명 | 리스크 수준 |
|--------|------|-----------|
| Slack Socket Mode | 무료 플랜에서 단일 WebSocket 연결 유지 필요 | 중간 (연결 끊김 시 재연결 로직 필수) |
| OpenRouter 무료 모델 | nex-agi/nex-n2-pro:free 모델의 가용성 및 속도 | 높음 (무료 모델 서비스 중단 가능성) |
| pgvector | PostgreSQL 확장 설치 필요 | 낮음 (Docker 이미지로 관리) |
| sentence-transformers | 로컬 임베딩 모델 크기 및 초기 로딩 시간 | 낮음 (컨테이너 빌드 시 캐싱) |

### 외부 의존성

| 의존성 | 설명 |
|--------|------|
| Slack API Rate Limit | `conversations.history` API: 50회/분 제한. 백필 배치 시 속도 조절 필요. |
| OpenRouter API 가용성 | 무료 모델 특성상 응답 지연 또는 일시 중단 가능성 있음. |
| Slack Workspace Admin 권한 | Slack App 생성 및 채널 `channels:history` 스코프 승인 필요. |

### 주요 리스크 및 완화 전략

| 리스크 | 발생 가능성 | 영향도 | 완화 전략 |
|--------|------------|--------|---------|
| 무료 OpenRouter 모델 중단 또는 품질 저하 | 중간 | 높음 | 모델명을 `properties.txt`로 분리하여 코드 변경 없이 모델 교체 가능하도록 설계. 백업 모델 목록 관리. |
| Slack 이벤트 중복 수신 | 낮음 | 낮음 | `event_id` 기준 DB unique 제약으로 멱등성 보장. |
| LLM 답변 환각(Hallucination)으로 잘못된 정보 전달 | 중간 | 높음 | Fallback 프롬프트로 불확실 답변을 담당자 호출로 전환. 사용자에게 "AI 생성 답변" 명시. |
| pgvector 검색 성능 저하 (대용량 임베딩) | 낮음 | 중간 | `ivfflat` 또는 `hnsw` 인덱스 적용. 청크 크기 및 상위 K 개수 조정. |
| 개인정보 유출 | 낮음 | 매우 높음 | 저장 전 PII 마스킹 필수 적용. DM 채널 수집 금지. |
| Slack Workspace Admin 승인 지연 | 중간 | 중간 | Phase 0 시작 시 Admin 승인 요청을 선제적으로 진행. |
