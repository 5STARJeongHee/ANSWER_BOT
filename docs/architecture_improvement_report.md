# 슬랙 봇 아키텍처 개선 완료 보고서

요청해주신 서버 환경(CPU 추론, 제한된 컨텍스트 길이)을 고려하여 슬랙 봇의 응답 파이프라인 최적화 작업을 모두 완료했습니다.

## 🛠️ 주요 변경 사항 요약

### 1. 스레드 문맥(Thread Context) 기반 인지 파이프라인
*   **`slack_bot/services/slack_service.py`**: Slack API `conversations.replies`를 호출하는 `fetch_thread_history` 함수를 추가하여 스레드 내 과거 대화 내역을 조회할 수 있도록 변경했습니다.
*   **`slack_bot/services/summarizer.py`**: `summarize_thread_context` 함수를 신규 생성했습니다. 스레드의 이전 문맥을 전부 LLM에 넣으면 컨텍스트 한계(170GB 메모리를 공유하는 CPU 서버의 제약)를 넘을 수 있으므로, 이전 대화를 2~3문장으로 짧게 요약하는 단계를 거칩니다.
*   **`slack_bot/handlers/event_handler.py`**: 스레드에서 질문이 인입될 경우, 위 과정(스레드 메시지 수집 -> 요약)을 거친 후 RAG 검색 및 메인 LLM 프롬프트에 **[스레드 이전 문맥 요약]** 정보로 주입되도록 파이프라인을 수정했습니다. 이제 대명사나 꼬리무는 질문에도 정확한 의도를 파악할 수 있습니다.

### 2. 질문 캐싱 및 외부 검색 우회 (Web Search Skip)
*   **`slack_bot/services/context_retriever.py`**: 스레드 요약본을 주입받아 RAG 검색 쿼리(`_generate_rag_query`)를 생성하도록 개선하여, 검색의 정확도를 높였습니다.
*   **`slack_bot/handlers/event_handler.py`**: 이전 답변 캐싱 방식을 "답변 그대로 반환"이 아닌 "다듬어서 반환"하는 방식으로 구현했습니다. RAG 검색에서 **유사도가 0.90(90%) 이상인 최상위 컨텍스트(과거 동일/유사 질문에 대한 봇의 응답)**가 발견되면, 느리고 비싼 웹 검색(Web Search) 파이프라인을 가동하지 않고 즉시 **기존 컨텍스트를 기반으로 LLM이 자연스럽게 재답변**하도록 최적화했습니다.

## ✅ 기대 효과
1.  **컨텍스트 이해도 대폭 상승**: "그거 다시 설명해줘", "A말고 B는 어때?" 같은 꼬리무는 질문 처리 가능.
2.  **응답 속도 개선**: 과거 자주 묻는 질문(FAQ)의 경우 무거운 웹 검색 단계를 건너뛰기 때문에 처리 시간이 크게 단축됩니다.
3.  **서버 리소스 절약**: 불필요한 전체 대화 원문 전달 및 웹 검색 후처리 작업을 줄여 공유 중인 서버 리소스(CPU/RAM) 점유율을 최적화했습니다.

## 🧪 검증 계획 (수동 확인 필요)
실제 봇을 재가동하시고 다음 두 가지 시나리오를 슬랙에서 테스트해 보시기를 권장드립니다:
1.  **스레드 꼬리 물기**: 특정 질문을 한 후, 봇의 답변 스레드에 "그 방법에 대해 더 자세히 설명해줘" 라고 대명사를 사용해 재질문.
2.  **캐싱 우회 확인**: 어제 답변했던 완벽히 동일한 질문을 새 스레드에 질문했을 때, 평소보다 훨씬 빠르고 웹 검색 없이 답변이 생성되는지 콘솔 로그(유사도 0.90 발견 로그) 확인.
